Kotlin Flow Operators: combine vs merge vs zip

9 May 2026

A visual deep-dive into the three Kotlin Flow operators for combining streams — what each one does, when to use it, and the gotchas to watch for. With marble diagrams and real-world examples.

When you start working with Kotlin Flows, you'll quickly run into situations where you want to combine values from multiple flows into one. The standard library gives you three operators for that — combine, merge, and zip — and they look interchangeable at first glance. They're not. Picking the wrong one is one of the most common sources of subtle reactive bugs in Android and server-side codebases alike.

This post walks through what each operator does, how to visualize it, and when to reach for which.

A quick refresher on Flow

A Flow<T> is Kotlin's coroutine-friendly way of representing a stream of values produced asynchronously over time. Conceptually it's Sequence<T> plus suspending functions: it can pause, resume, and respect coroutine cancellation. You build a flow with flow { emit(...) }, flowOf(...), or by converting from other types (asFlow(), callbackFlow {}, etc.), and you consume it from a coroutine with collect { }.

Most non-trivial code involves multiple flows. A UI screen might have one StateFlow for the search query and another for the active filter. A repository might emit both network and cache events. Sooner or later you need to merge them — and that's where the three operators come in.

In each marble diagram below, flow1 emits numbers and flow2 emits letters at different rates. Watch where the result emissions land on the timeline.

Zip — pair values 1-to-1

zip waits for both flows to emit, then pairs the values and emits the pair (or the result of a transform). The faster flow stalls — its emissions are buffered until the slower flow catches up. The result completes as soon as the shorter of the two flows ends; any leftover values from the longer flow are dropped.

val nums = flowOf(1, 2, 3, 4).onEach { delay(100) }
val letters = flowOf("A", "B", "C").onEach { delay(300) }

nums.zip(letters) { n, l -> "$n$l" }.collect(::println)
// Output:
// 1A   (after ~300ms)
// 2B   (after ~600ms)
// 3C   (after ~900ms)
// 4 is never emitted because letters has no fourth value

When to use it

  • Pairing items from two parallel sequences that match up in order: file uploads with their checksums, requests with their responses, two halves of a calculation.
  • Any time the position in each flow matters and you want strict 1-to-1 alignment.

Gotchas

  • It's almost never what you want for state. If query and filter are both StateFlows representing UI state, you do not want zip. It will pair the first query with the first filter, the second with the second, and so on — even if the user only changed one of them. Use combine for that.
  • The faster flow gets backpressured. zip suspends emissions while waiting for the other side, so don't use it when one of the flows is hot and expected to be lossy.

Combine — emit on every change with the latest values

combine emits whenever any flow emits, using the latest value from each input flow. It needs every flow to have emitted at least once before it produces anything — that's why there's no result emission until both 1 and A have arrived in the diagram above.

val nums = flowOf(1, 2, 3).onEach { delay(150) }
val letters = flowOf("A", "B", "C").onEach { delay(150) }

combine(nums, letters) { n, l -> "$n$l" }.collect(::println)
// Output (timing-dependent):
// 1A
// 2A
// 2B
// 3B
// 3C

When to use it

This is the workhorse for reactive state. Anywhere you have multiple sources of state and want a derived result that updates whenever any source changes:

data class SearchUiState(
    val query: String,
    val filter: Filter,
    val results: List<Item>
)

val uiState: Flow<SearchUiState> = combine(
    queryFlow,
    filterFlow,
    resultsFlow
) { query, filter, results ->
    SearchUiState(query, filter, results)
}

Gotchas

  • Silent until all sources emit. If one of your input flows takes a long time to produce its first value (e.g. waits on a network call), the combined flow stays silent. Provide a sensible initial value with onStart { emit(default) } upstream, or use a StateFlow which always has one.
  • Not synchronized. combine doesn't wait for all flows to emit something new — it emits as soon as any one of them does, even if the others' values haven't changed since the last emission. That's usually the point, but make sure your downstream is idempotent.
  • Variants. combine has overloads for 2 to 5 flows. For more, pass a vararg of flows or a List. There's also combineTransform, which gives you a FlowCollector so you can emit zero or many times per change.

Merge — interleave into one stream

merge collects from all input flows concurrently and re-emits values as they arrive, preserving original order within each flow but interleaving freely between them. There's no transformation lambda — same type in, same type out. The result completes when all input flows have completed.

val flow1 = flowOf(1, 2, 3).onEach { delay(100) }
val flow2 = flowOf(10, 20, 30).onEach { delay(150) }

merge(flow1, flow2).collect(::println)
// Output (interleaved by arrival time):
// 1, 10, 2, 3, 20, 30

When to use it

You have multiple sources of the same kind of event and want all of them: clicks from different buttons, messages from different channels, pushes from multiple WebSockets.

val allUserActions: Flow<UserAction> = merge(
    homeButton.clicks,
    settingsButton.clicks,
    backGesture.events
)

Gotchas

  • Order between flows is non-deterministic. Don't rely on flow1 winning a tie — it depends on the dispatcher and scheduling.
  • Concurrent collection. Each input flow is collected on its own coroutine. If your producers do thread-confined work, be aware.
  • For Flow<Flow<T>>, look at flattenMerge (and its siblings flattenConcat, flatMapMerge, flatMapConcat, flatMapLatest).

A real-world example: reactive search

Here's a snippet that uses both combine and merge in the service of one feature — a search screen that re-queries an API based on the latest query and filter, and re-queries when the user taps a refresh button:

class SearchViewModel(
    private val repository: SearchRepository
) : ViewModel() {

    private val queryFlow = MutableStateFlow("")
    private val filterFlow = MutableStateFlow(Filter.All)
    private val refreshClicks = MutableSharedFlow<Unit>()

    private val searchTriggers: Flow<Pair<String, Filter>> =
        combine(queryFlow, filterFlow) { q, f -> q to f }

    val results: Flow<List<Item>> = merge(
        searchTriggers,
        refreshClicks.map { queryFlow.value to filterFlow.value }
    )
        .debounce(300)
        .flatMapLatest { (q, f) -> repository.search(q, f) }
}

zip doesn't fit here because the query and filter are independent UI state, not paired sequences. It would, however, be the right call if you were fetching a list of user IDs and, in parallel, loading each one's avatar in matching order:

val users: Flow<User> = userIdFlow.map { repo.getUser(it) }
val avatars: Flow<Bitmap> = userIdFlow.map { repo.getAvatar(it) }

val cards: Flow<UserCard> = users.zip(avatars) { user, avatar ->
    UserCard(user, avatar)
}

Both flows are derived from the same indexed sequence, so pairing strictly 1-to-1 is exactly what you want.

TL;DR — three mental models

  • zip = a zipper. Matches teeth, drops extras.
  • combine = a spreadsheet cell. Recalculates whenever any input changes.
  • merge = a postal sorter. Many bins → one belt.

If you keep those three pictures in your head, you'll always reach for the right one.
